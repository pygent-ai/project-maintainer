# Project Maintainer Artifact Structure

Create the delivery artifact at the target repository root:

```text
.doc_project_maintainer/
  README.md
  INDEX.md
  manifest.yaml
  project/
    overview.md
    architecture.md
    glossary.md
    build-plan.md
    source-symbol-inventory.json
    coverage-map.json
    open-questions.md
    flows/
      <flow-id>.md
  modules/
    <module-id>/
      README.md
      design.md
      directories.md
      dependencies.md
      changes.md
      git-history.md
  directories/
    <encoded-path>/
      README.md
      files.md
      module-links.md
      changes.md
  code/
    <source-relative-path>/
      <source-file>/
        <source-file>.md
        Class <ClassName>/
          Class <ClassName>.md
          <ClassName>.<methodName>.md
          <ClassName>.<methodName>/
            actual-behavior.md
            contracts.md
            side-effects.md
            health.md
            risks.md
            tests.md
        <functionName>.md
        <functionName>/
          actual-behavior.md
          contracts.md
          side-effects.md
          health.md
          risks.md
          tests.md
  decisions/
    ADR-0001-short-slug.md
  changes/
    records/
      CHG-YYYYMMDD-NNN-short-slug.md
    by-module/
      <module-id>.md
    by-directory/
      <encoded-path>.md
```

## Required Root Files

- `README.md`: Project brief, current documentation status, technical stack, startup/test hints, and maintenance rules.
- `INDEX.md`: Human and agent navigation for modules, directories, flows, code symbols, changes, and decisions. Keep this as links and one-line summaries only.
- `manifest.yaml`: Machine-readable relationship graph for modules, directories, flows, code symbols, changes, and decisions.
- `project/build-plan.md`: Coverage plan for large or partially mapped projects. Track completed slices, pending slices, stale areas, pending flow slices, pending code symbol slices, coverage closure audit results, and suggested next work.
- `project/source-symbol-inventory.json`: Machine-readable source symbol inventory generated by `scripts/inventory_symbols.py`, including extractor, confidence, file hash, function and method symbols, and doc verification status.
- `project/coverage-map.json`: Machine-readable progress ledger generated by `scripts/inventory_symbols.py`, including git head, dirty worktree status, untracked candidate source files, source hashes, per-file coverage status, per-symbol doc health status, removed files, and suggested subagent slices.

## Code Symbol Docs

Use `code/` for executable-detail docs that mirror source paths. Do not use `README.md` inside `code/`. File names must carry symbol identity:

- source file docs: `<source-file>/<source-file>.md`, such as `a.py/a.py.md`
- class docs: `Class <ClassName>/Class <ClassName>.md`
- method entry docs: `Class <ClassName>/<ClassName>.<methodName>.md`
- method detail docs: `Class <ClassName>/<ClassName>.<methodName>/<detail>.md`
- module-level function entry docs: `<functionName>.md`
- module-level function detail docs: `<functionName>/<detail>.md`

Use `references/code-symbol-docs.md` for the full naming, template, health, and coverage rules.

Every stable source file needs a file-level code doc with a symbol inventory. Every top-level function and every method on every top-level class needs an entry doc with `Actual Role` and health summary fields before code symbol coverage can be `current`.

Run `scripts/inventory_symbols.py` through its single command entry point and write both `project/source-symbol-inventory.json` and `project/coverage-map.json`. It automatically chooses `python_ast`, optional `ctags`, or dependency-free `heuristic` fallback per file. Heuristic or `requires_review` results can scaffold docs but keep code symbol coverage `partial` until reviewed, re-run with a stronger extractor, or marked out of scope.

## Size Budgets

Use byte size as the enforceable limit. Token count varies by language and content type.

- `INDEX.md`: 5 KB
- root `README.md`: 8 KB
- `manifest.yaml`: 20 KB
- `project/*.md`: 8 KB each
- `project/source-symbol-inventory.json`: 256 KB; split or compress only if the repository is very large and keep a small summary in `project/build-plan.md`
- `project/coverage-map.json`: 256 KB; for very large repositories keep this as the authoritative machine-readable ledger and summarize only the active slices in `project/build-plan.md`
- `project/flows/*.md`: 8 KB each
- `modules/*/README.md`: 8 KB
- `modules/*/*.md`: 8 KB each
- `directories/*/README.md`: 6 KB
- `directories/*/*.md`: 6 KB each
- `code/**/*` symbol entry docs: 4 KB each
- `code/**/*` file-level, class-level, and detail docs: 6 KB each
- `changes/records/*.md`: 6 KB each
- `changes/by-*/*.md`: 10 KB each
- `decisions/*.md`: 6 KB each

When a file exceeds budget, split by topic, time period, or submodule. Leave a short summary and links in the parent file.

## Naming

Use stable IDs for records:

- Changes: `CHG-YYYYMMDD-NNN-short-slug.md`
- Decisions: `ADR-NNNN-short-slug.md`
- Modules: lowercase kebab-case IDs
- Flows: lowercase kebab-case IDs that name the causal behavior, stored as `project/flows/<flow-id>.md`
- Encoded paths: replace path separators with double underscores
- Code symbols: preserve source names in path and file names wherever the filesystem permits. Use `<ClassName>.<methodName>.md` for methods and `<functionName>.md` for module-level functions. If a symbol name collides, append a minimal stable disambiguator while keeping the readable symbol prefix.

Use semantic filenames for change records. Do not use date-only filenames because agents cannot infer relevance from them.

## Linking Model

Use three link layers:

1. Markdown links for humans.
2. YAML frontmatter in each record for stable IDs and relationships.
3. `manifest.yaml` as the central relationship graph.

Modules link to:

- owning directories
- related flows
- related changes
- related decisions
- tests and verification commands

Directories link to:

- mapped modules
- related flows
- related code symbols
- notable files
- related changes
- related decisions

Flows link to:

- affected modules
- affected directories
- affected code symbols when producer, consumer, state mutation, contract, or output behavior depends on specific functions or methods
- source-of-truth, cache, generated, telemetry, debug, fixture, transient, or local-state paths when applicable
- entrypoints, producers, consumers, output surfaces, and verification commands
- related changes and decisions

Code symbols link to:

- owning source file, class, function, or method path
- affected modules and directories
- related flows when the symbol participates in causal behavior
- related changes, decisions, and tests
- detail docs for actual behavior, contracts, side effects, health, risks, and tests

Changes link to:

- affected modules
- affected directories
- affected flows
- affected code symbols
- decisions that explain the change
- commits when git data exists

Decisions link to:

- affected modules
- affected directories
- affected flows
- affected code symbols
- changes that implemented or revised the decision

## Sync And Coverage State

Every existing artifact needs a preflight sync assessment before use. Store durable state in `manifest.yaml` and human-readable status in `project/build-plan.md`.

Track at least:

- artifact version
- last artifact update date
- last scanned commit when git exists
- last synced commit when git exists
- sync status: `current`, `partial`, `stale`, or `unknown`
- coverage status by module, directory, flow, and code symbol
- coverage map path, generation time, git head, dirty worktree state, stale files, and suggested slices
- coverage closure audit status, source, and last audit date
- pending sync items with affected paths, modules, directories, flows, code symbols, and reason

Use `project/coverage-map.json` as the machine-readable source of truth for per-file and per-symbol progress. Use `project/build-plan.md` for the human-readable execution plan in large projects, partial migrations, and multi-agent work. Keep it short and operational: what is done, what is trusted, what remains, what is blocked, and what module, directory, flow, or code symbol slice should be built next.

Only use `current` for project-wide sync or coverage status after a coverage closure audit has run, no actionable pending slices remain, no undocumented top-level functions or class methods remain, and any explicit skip reason has been recorded. For git repositories, the audit source should be `git ls-files` plus `git status --short`. For non-git repositories, record the substitute listing source. The audit proves directory/module coverage by stable boundaries and code symbol coverage by exhaustive source symbol inventory.

Before marking project-wide coverage `current`, every stable tracked path must be one of:

- mapped to a module or directory entry,
- marked out of scope with a reason.

Pending slices are allowed and useful for partial coverage plans, but they keep project-wide coverage `partial` until completed or marked out of scope.

Untracked paths must be classified as artifact output, local runtime state, generated output, or candidate project files. Candidate project files keep coverage partial until they are mapped, ignored, or marked out of scope. Generated, cache, runtime, session, coverage-report, virtual-environment, and local-state paths should be excluded intentionally rather than silently omitted.

Before marking flow coverage `current`, every documented user-visible, stateful, generated, integration, background, or event-driven behavior must have a completed flow doc or an explicit out-of-scope reason. Pending flow slices keep flow coverage `partial`. Flow coverage proves causal traceability, not exhaustive line-level behavior.

Before marking code symbol coverage `current`, every stable source file must have a file-level code doc and symbol inventory, every top-level function must have an entry doc with `Actual Role` and health summary fields, and every method on every top-level class must have an entry doc with `Actual Role` and health summary fields. Explicit out-of-scope reasons may exclude generated, vendored, build output, disposable local state, or unsupported files. Pending code symbol slices keep code symbol coverage `partial`. Code symbol coverage proves queryable source-level traceability and health disposition for every required top-level function and class method, not exhaustive commentary on every local branch.

Status meanings:

- `planned`: Documentation structure or coverage is intended but not yet created.
- `partial`: Some stable areas are mapped, but known gaps, pending slices, stale areas, or unclassified paths remain.
- `scanned`: A directory or slice has been inspected and has evidence, but project-wide closure is not implied.
- `current`: The artifact is synced to the recorded source, stable tracked paths are mapped or out of scope, every required source symbol has an entry doc with health, and no actionable pending slices remain.
- `stale`: The artifact is known to lag behind code, docs, git history, or project structure.
- `unknown`: The artifact has not been assessed.

When multiple agents work on the artifact, split work by module or directory. Do not let agents edit the same module or directory doc concurrently unless a coordinator merges the outputs.

For initial scans with many files or symbols, assign work from `coverage-map.json` `suggested_slices`. The coordinator must update the map after each integrated slice, compare source hashes against git, classify untracked candidate project files, and keep stale or removed files explicit until they are synced or dispositioned.

## Confidence Rules

Use one of:

- `confirmed`: Supported by code, tests, docs, commit messages, PRs, or issues.
- `inferred`: Reasonable interpretation from structure or code behavior.
- `unknown`: Evidence is insufficient.

Never present inferred rationale as fact.
